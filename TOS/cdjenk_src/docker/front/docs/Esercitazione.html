Build dell’applicazione

  • La compilazione e la build può essere effettuata con maven

  • Gli elementi principali di Maven sono:

    • Il comando: mvn

    • Il file pom.xml che descrive le caratteristiche del progetto per la build

Note
 POM = Project Object Modeling

  • Maven implementa il processo di build tramite comandi chiamati Goals

  • Un insieme di goals predefiniti permettono di eseguire il processo di build utilizzando le configurazioni presenti nel file pom.xml

  • Nel pom.xml possono essere configurate delle azioni da fare prima e/o dopo l’esecuzione dei goal standard

Note
 Nell’esercitazione verranno utilizzati solo i goal standard

  • Il pom.xml riporta le dipendenze dell’applicazione che verranno scaricate da maven durante il processo di build

  • di default maven salva le dipendenze nella cartella ~/.m2/repository

  • Eseguire i seguenti comandi per avviare il processo di build:

$ cd demoapp/
$ mvn clean install
...

  • Se tutto funzionerà correttamente, il comando terminerà con il messaggio "Build Success" :

Esecuzione dell’applicazione

L’artefatto generato dall’applicazione è un jar standalone.

Note
 jar = "Java Archive". E' un archivio zip che contiene le classi Java compilate e i metadata

Questo jar contiene al suo interno un embedded application server. Per questo l’esecuzione dell’applicazione non richiede un Java application server come Tomcat o Jetty

Note
 Questo tipo di jar è conosciuto anche come "Uber-JAR"

L’applicazione verrà eseguita in 2 modi:

  • On the metal: eseguendo l’embedded application server "as-is".

  • Tramite Docker: build di una Docker image ed esecuzione di un Docker Container

Esecuzione con Docker

Eseguire l’applicazione con Docker permette di:

  • Avere una migliore portabilità. Nel Dockerfile andiamo a specificare tutto il necessario per eseguire l’applicazione.

  • Sfruttando l’ambiente del laboratorio, che si basa su Docker, sarà possibile verificare l’applicazione da un web-browser.

Note
 Negli esercizi del laboratorio si utilizzeranno solo alcuni comandi di Docker. Per maggiori informazioni relative a Docker leggere la documentazione ufficiale!

Il workflow di "Docker" prevede:

  • Tramite un Dockerfile, anremo a creare una "Docker image" per l’applicazione

    • Il Dockerfile è un file di testo che permette di specificare la ricetta per eseguire un’applicazione (singolo processo)

    • Una "Docker image" è un template immutable

    • Dalla "Docker image" verrà istanziato un "Docker container"

    • Docker ci permette di gestire il port forwarding per permettere di accedere all’applicazione da un web-browser.

Tip
 Un Dockerfile è presente nella cartella principale del progetto? Andiamo ad utilizzarlo per avviare l’applicazione!

  • Comandi da eseguire :

# From the demoapp folder
$ docker build -t "demo-app:1.0.0" ./
...
Successfully built 8e7e2115adb8
# Nice, now launching the container
$ docker run -d --name demoapp-1 -p 30000:8080 demo-app:1.0.0

  • -d esecuzione in background (detached mode)

  • -p 30000:8080 pubblica la porta 8080 del container nella porta esterna 30000.

  • Per vedere l’applicazione andare su: http://localhost:30000/

  • Ricaricare la pagina per vedere i cambiamenti

Eseguire la build e gestire le failures

L’obiettivo dell’esercizio è eseguire la build e i test con successo ed esplorare le funzionalità dei job e lo step relativo alle builds.

Fare fallire la Build

L’idea è usare un approccio conservativo, quando gli Integration test segnalano degli errori far fallire la build (al posto di rendere la build instabile).

  • Accedere alla configurazione demoapp-build,

  • Andare nella Build section

  • Cliccare sul bottone Advanced

  • Aggiungere il parametro -Dmaven.test.failure.ignore=false nel campo MAVEN_OPTS

  • Salvare ed eseguire la build che fallirà:

Correggere la Build

Come sviluppatore, l’obiettivo è trovare la ragione del fallimento dei test e correggerli.

  • Accedere alla sezione Latest Test Result dalla Job Page

  • Nella pagina Test Result, scoprire gli Error Details cliccando sull’icona +, per verificare la causa delle failure:

Errori

java.net.BindException: Address already in use
By default, the \ is started for the Integration Tests on the 8080 port

Jenkins is already listening on the 8080 port

  • Verificare il messaggio usando la Devbox:

$ curl -I http://jenkins:8080
HTTP/1.1 404 Not Found
Cache-Control: must-revalidate,no-cache,no-store
Content-Type: text/html;charset=iso-8859-1
Content-Length: 437
Server: Jetty(9.4.30.v20200611)
# Jetty Server is responding to HTTP protocol: port is already used

  • Per correggere il problema, è necessario configurare gli Integration test in modo da usare una porta disponibile. Ad esempio la porta 8082

  • Accedere alla gitserver come butler e modificare il file src/test/resources/helloworld-integrationtest.yaml nel seguente modo

template: Hello, %s!
defaultName: IntegrationTest
server:
  rootPath: /api/
  applicationConnectors:
  - type: http
    port: 8082

Continuous Delivery con Jenkins

Progettazione di una CD Pipeline

L’obiettivo dell’esercizio è di avere una chiara comprensione delle CD Pipeline utilizzate nelle successive esercitazioni.

Ambiente

L’applicazione demo è composta di 4 ambienti:

  • Development Environment:

    • Rappresenta l’ambiente di sviluppo

      • Devbox + WebIDE

  • Forge Environment:

    • L’ambiente centralizzato dove eseguire le build e test dell’applicazione.

      • Jenkins + GitServer + Artifactory + Docker registry

      • Docker containers su porte >= 32768

  • Staging Environment:

    • L’ambiente di staging è lo specchio di production, usato per validare le applicazioni

      • Docker containers su porte [20000:30000]

  • Production Environment

    • L’ambiente del cliente finale.

      • Docker containers su porte 10000

    • Questo ambiente potrebbe contenere altri servizi non contemplati nell’esercitazione.

User Profiles

Sono definiti 3 profili:

  • End-Users: Utenti finali che usano l’applicazione in produzione

  • Product Managers: Figure che validano l’applicazione prima di rilasciarla agli End-Users

  • Engineers: Sviluppatori, sysadmin, etc. che sviluppano e mantengono l’applicazione

Pipeline proposta

Questo schema rappresenta la CD pipeline proposta

  • Dettaglio degli stages:

    • Checkout: Ottenimento del codice sorgente. Questo step può andare in errore a causa di problemi di rete

      • Feedback: quando è in fallimento questo step manda un feedback agli Engineers

    • Build: dal dato codice sorgente, esegue la compilazione e la costruzione dell’artefatto

      • Feedback: quando fallisce provvede ad avvisare gli Engineers

    • Unit Tests: Lancia una serie di Unit Tests che devono essere veloci

      • Feedback: quando fallisce provvede ad avvisare gli Engineers

    • Smoke & Acceptance Tests: Quando si superano gli Unit Tests lancia gli smoke e acceptance tests per completezza.

      • Feedback: Quando fallisce provvede ad avvisare sia gli Engineers che Product Managers

    • Staging Deployment: se si sono superati tutti i test, si procede col rilascio automatico nello Staging environment.

      • Feedback: In caso di successo viene mandata una notifica ai Product Managers con l’url dell’applicazione, se invece è fallita la notifica viene mandata agli Engineers

    • Production Deployment Request: a partire dall’applicazione in Staging, se i Product Managers approvano il rilascio, viene mandata una notifica agli Engineers per richiedere il Production Deployment

      • Feedback: gli step stessi.

    • Production Deployment: A partire dalla Production Deployment Request, gli Engineers possono promuovere la build per il rilascio in produzione.

      • Feedback: Tutti sono notificati del prossimo rilascio.

Esercizio completato!

Implementazione di una CD pipeline con Jenkins

L’obiettivo dell’esercitazione è definire una CD pipeline con Jenkins in modo da implementare il processo definito precedentemente.

Il processo che vogliamo ottenere è il seguente:

Il processo è composto da 3 job:

  • demoapp-build: job di build che:

    • scarica i sorgenti dal VCS

    • effettua i test e crea l’artefatto

    • innesca l’esecuzione del job demoapp-docker-builder passando il parametro APP_VERSION

  • demoapp-docker-builder: job di build che:

    • scarica gli artefatti prodotti dal processo demoapp-build

    • effettua la build docker e crea l’immagine dell’applicazione

    • pubblica l’immagine docker nel registry

    • innesca l’esecuzione del job demoapp-staging-deployer passando il parametro DOCKER_IMAGE

  • demoapp-staging-deployer: job che:

Polling must die: Triggering with Hooks

Polling SCM o eseguire la build in modo schedulato non è una buona pratica:

  • Dal punto di vista del sistema: spreco di risorse, esecuzioni non necessarie

  • Dal punto di vista dello sviluppatore: Il tempo tra il codice committed/pushed e la build potrebbe essere troppo lungo.

La soluzione è usare gli Hook: Un hook è una richiesta di build che viene innescata da un sistema esterno (per esempio quando avviene un determinato evento nel VCS).

Gogs, il GitServer, mette a disposizione un WebHook che permette di fare richieste HTTP quando accadono degli eventi sul server Git (per esempio una Push).

  • Installare il plugin senza richiedere il riavvio

Da una CD Pipeline a una Jenkins Pipeline

Obiettivo dell’esercizio è passare da una CD pipeline a una completa Jenkins Pipeline.

Creare la Pipeline

  • Creare un nuovo Pipeline Job, chiamarlo demoapp-cd-pipeline

  • Fare in modo che usi il git repository come Pipeline source (Pipeline script from SCM)

  • con la WebIde inserire il seguente contenuto nel Jenkinsfile

Tip
 Il contenuto sarà delineato stage by stage 
node {
  stage 'Checkout'
  checkout scm

  stage 'Configuration'
  sh 'git rev-parse --short HEAD > GIT_COMMIT'
  short_commit = readFile('GIT_COMMIT').trim()
  sh 'echo localhost > PUBLIC_IP'
  def pom = readMavenPom file:'pom.xml'
  artefactName = "${pom.getArtifactId()}.${pom.getPackaging()}"
  publicIp = readFile('PUBLIC_IP').trim()
  def mvnHome = tool name: 'maven3', type: 'hudson.tasks.Maven$MavenInstallation'
  sh 'rm GIT_COMMIT PUBLIC_IP'

  stage 'Build'
  sh "${mvnHome}/bin/mvn clean compile"

  stage 'Unit Tests'
  sh "${mvnHome}/bin/mvn test"
  step([$class: 'JUnitResultArchiver', testResults: 'target/surefire-reports/*.xml'])

  stage 'Package'
  sh "${mvnHome}/bin/mvn package"
  stash name: 'app', includes: "target/${artefactName}"
  stash name: 'dockerfile', includes: "Dockerfile"
  stash name: 'config', includes: "hello-world.yml"

  stage 'Integration Tests'
  sh "${mvnHome}/bin/mvn verify"
}

node('docker'){
  stage 'Docker build'
  unstash 'config'
  unstash 'dockerfile'
  unstash 'app'
  dockerImage = "demo-app:${short_commit}"
  image = docker.build("${dockerImage}")
  docker.withRegistry('http://localhost:5000', '') {
    image.push("${short_commit}")
  }

  stage 'Smoke Test'
  container = image.run()
  container.stop()
}

stage 'Deploying to staging'
build job: 'demoapp-staging-deployer', parameters: [[
  $class: 'StringParameterValue',
  name: 'DOCKER_IMAGE',
  value: "localhost:5000/${dockerImage}"
]]

timeout(time:5, unit:'DAYS') {
  input message: "http://${publicIp}:20000. Is it ok?", ok: 'Production delivery'
}

node('docker') {
  stage 'Production Delivery'
  echo "Dummy step simulating Production Delivery"
